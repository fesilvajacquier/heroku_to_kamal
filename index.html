Contexto

SafetyPanel

Arrancó en 2020 como una app de capacitaciones encargada a una software factory.

El dueño tiene una consultora de higiene y seguridad. Con el tiempo, sus requerimientos de negocio la transformaron en una plataforma completa de gestión de seguridad y compliance.

HyS = Higiene y Seguridad (occupational health & safety). En Argentina, las empresas están obligadas por ley a tener un servicio de HyS. SafetyPanel ayuda a gestionar capacitaciones, visitas, documentación, y cumplimiento normativo.

A medida que la app resolvía más problemas, empezaron a dejar servicios de terceros.

Después, colegas, otras consultoras, incluso sus propios clientes empezaron a preguntar:

"¿Podemos usar esto nosotros también?"

Hoy tiene dos tipos de clientes:

Consultoras de HyS — como el dueño original

Empresas — fábricas, plantas, operaciones con sus propias necesidades de seguridad

En varios países de Sudamérica.

Multi-tenant, basada en subdominios

Cada cliente tiene acme.safetypanel.com.ar

acts_as_tenant

acts_as_tenant es una gem que agrega scoping automático a todos los queries por tenant. Se configura con set_current_tenant_through_filter y usa el subdominio del request para identificar al tenant. Cada query se scopea automáticamente — no podés acceder a datos de otro tenant por error.

Motivación

¿Por qué dejar Heroku?

SSL wildcard

Automatizar certificados wildcard en Heroku era imposible o muy caro.

Tenía que renovar certificados manualmente cada 3 meses.

Un certificado wildcard (*.safetypanel.com.ar) cubre todos los subdominios con un solo cert. Let's Encrypt los da gratis pero vencen cada 90 días. En Heroku, la renovación automática con ACM solo funciona para dominios individuales, no wildcard. La alternativa era el add-on SSL de Heroku que costaba bastante o renovar manualmente via CLI cada 3 meses.

Cosas para modernizar

wkhtmltopdf estaba sin mantenimiento. Las alternativas son Chrome-based.

Chrome necesita más memoria → dynos más grandes → más plata.

Las alternativas modernas a wkhtmltopdf (Grover, ferrum_pdf, Puppeteer) todas usan Chrome/Chromium headless para renderizar. Chromium necesita ~100-200 MB de RAM por instancia. En un dyno de 512 MB que también corre Puma, esto fuerza un upgrade a 1 GB ($50/mes).

El costo iba a ser un problema

La app estaba creciendo en tráfico y clientes.

Escalar en Heroku es caro. Todavía no era un problema — pero no queríamos llegar a ese punto.

Un dyno Standard-1X (512 MB) cuesta ~$25/mes. Standard-2X (1 GB) ~$50/mes. Agregar Redis (para Sidekiq) son otros ~$15/mes mínimo. Managed Postgres desde ~$9/mes. Para una app chica puede parecer poco, pero escalar horizontalmente (más dynos) o verticalmente (más RAM) se pone caro rápido comparado con un VPS donde controlás los recursos.

Pensamos crecer

Muchas más features por delante. Muchos más clientes.

Necesitábamos modernizar el stack primero para que ese camino fuera más fácil.

El One Person Framework

Rails está apostando al framework para una persona. Esta app la desarrolla y mantiene una persona.

Todo lo que se aparta del camino estándar de Rails es algo más para mantener.

Quería volver al camino principal: los defaults de Rails 8 están pensados para esto.

Concepto de la keynote de DHH en Rails World 2024 (Toronto). Rails 8 viene con Solid Cache, Solid Queue, Kamal, Thruster, y Propshaft out of the box. La visión: un solo desarrollador puede tener una app en producción sin depender de servicios externos (Redis, Memcached) ni de PaaS caros. Apartarse del camino principal de Rails significa: más gems para mantener, más config custom, más cosas que pueden romperse en cada upgrade. Siguiendo los defaults de Rails 8, cada upgrade del framework mejora tu stack gratis. Kamal es el deployer oficial de Rails — es lo que reemplaza a Heroku en esta visión. Solid Cache y Solid Queue reemplazan Redis. Es exactamente lo que necesitábamos.

Las gems "Solid" son las nuevas dependencias de Rails 8 que reemplazan servicios externos por la base de datos. Solid Cache usa la DB como cache store (reemplaza Redis/Memcached). Solid Queue usa la DB para background jobs (reemplaza Sidekiq/Resque que necesitan Redis). El beneficio: PostgreSQL es la única dependencia externa.

Decisión clave

Upgradear Rails y cambiar las piezas internas
en Heroku primero

Después migrar la infraestructura.
El VPS recibe un stack limpio desde el día uno.

La idea es minimizar la cantidad de variables que cambian al mismo tiempo. Si upgradeamos Rails, cambiamos el queue backend, el cache y el motor de PDF todo en Heroku primero, cada cambio se puede testear en producción de forma aislada. Cuando el VPS recibe la app, ya tiene el stack final — menos cosas que pueden fallar en el día del deploy.

Feb 11

Framework Defaults + Motor de PDF

wicked_pdf → ferrum_pdf

#7 — el cambio de código más grande

Nuevas acciones "preview" — renderizan el PDF como HTML

PdfTokenAuthenticatable — tokens firmados, 30s de expiración

Chrome navega a la URL de preview y captura el PDF

ferrum_pdf usa la gem Ferrum (wrapper del Chrome DevTools Protocol) para controlar un Chromium headless. El flujo es: 1) la app renderiza el PDF como HTML en una acción "preview", 2) genera un token firmado con MessageVerifier (expira en 30 segundos), 3) Chromium navega a la URL de preview con ese token, 4) captura el PDF con page.pdf(). El concern PdfTokenAuthenticatable permite que la acción preview se autentique con el token firmado en vez de la sesión de usuario, porque Chromium headless no tiene cookies de sesión.

Primer problema en producción

843 MB

en un dyno de 512 MB

batch_size 500 → 100 · polling_interval 0.1s → 1s · jemalloc
Solid Queue hace polling a la DB para buscar jobs nuevos. Los defaults eran muy agresivos: batch_size 500 (cuántos jobs levanta por poll) y polling_interval 0.1s (cada cuánto pollea). Bajándolos se redujo mucho el uso de memoria. jemalloc es un memory allocator alternativo a glibc malloc. Reduce la fragmentación de memoria en Ruby significativamente (20-40% menos RSS). Se instala como buildpack en Heroku o como package en Docker.

Multi-stage Dockerfile: una etapa "build" que instala Node, compila assets y gems, y una etapa "runtime" que solo copia lo compilado. Esto reduce el tamaño de la imagen final significativamente. El script bin/provision-vps automatiza el hardening del servidor: firewall (ufw), fail2ban, SSH hardening (deshabilitar login por password, cambiar puerto), crear usuario deploy, instalar Docker, etc.
Ingredientes clave

jemalloc · YJIT · Chromium · Thruster

YJIT (Yet another Just-In-Time compiler) es el JIT compiler de Ruby, desarrollado por Shopify. Incluido en Ruby desde 3.1. Se activa con RUBY_YJIT_ENABLE=1. Mejora el throughput de la app un 15-25% sin cambios de código. Thruster es un HTTP/2 proxy de Basecamp escrito en Go. Se pone delante de Puma y maneja: compresión de assets (gzip/br), HTTP/2, X-Sendfile para servir archivos, y caching de assets estáticos. Viene por default en las apps Rails 8 nuevas.
Feb 24

Pero primero — probar en local

Antes de tocar producción, deployar a una VM local

La receta

Levantar una VM Ubuntu con Multipass

Agregar entradas en /etc/hosts apuntando subdominios a la IP de la VM

Configurar el browser para aceptar certs auto-firmados — thisisunsafe

Correr kamal setup apuntando a la VM

Multipass es una herramienta de Canonical para crear VMs Ubuntu de forma rápida. Con "multipass launch" tenés una VM corriendo en segundos. Ideal para simular un VPS local. El truco de "thisisunsafe": cuando Chrome muestra un error de certificado y NO te da la opción "Proceed anyway" (como pasa con HSTS), podés escribir "thisisunsafe" en cualquier parte de la página (no hay input visible, simplemente tecleás) y Chrome bypasea la verificación del certificado. Es un interstitial bypass keyword hardcodeado en Chromium. Se necesita porque la VM tiene un cert auto-firmado que Chrome rechaza.
Y ahí empezaron los tropiezos

Qué se rompió

Docker permission denied
El usuario no estaba en el grupo docker
Bloqueado de la VM
El hardening de seguridad fue demasiado agresivo — SSH rechazaba todas las conexiones
Login fallaba en silencio
Cert auto-firmado + interacción con Turbo — sin error, simplemente no pasaba nada
El problema del login silencioso: Turbo Drive intercepta la respuesta del form de login. Con un cert auto-firmado, el browser bloquea requests internos de Turbo sin mostrar error visible. El usuario hace click en "Login" y no pasa nada. La solución fue aceptar el cert (thisisunsafe) en todas las URLs que la app usa internamente. El lockout de SSH: el script de hardening cambiaba el puerto de SSH, deshabilitaba password auth, y configuraba fail2ban todo junto. Si algo fallaba en la config de sshd, SSH se reiniciaba con una config inválida y no podías volver a entrar.
500 en un formulario
Cannot get a signed_id for a new record
Los PDFs mostraban un 404
Chromium dentro de Docker no podía resolver las URLs de subdominios
El error "Cannot get a signed_id for a new record" ocurre con Active Storage cuando intentás obtener el signed_id de un blob que todavía no se guardó en la DB. Esto puede pasar si el formulario asume que el attachment ya fue persistido. El 404 de PDFs: ferrum_pdf levanta un Chromium headless dentro del container Docker. Cuando Chromium intenta navegar a "acme.safetypanel.com.ar/preview/...", esa URL no resuelve a nada dentro del container — el DNS público apunta al server externo, no al localhost del container.

El fix del PDF

Chromium dentro de Docker no puede resolver acme.safetypanel.com.ar


--host-resolver-rules "MAP *.safetypanel.com.ar <host-ip>"

KAMAL_HOST provee la IP del servidor

--ignore-certificate-errors — Chromium no confía en la CA de Cloudflare

--host-resolver-rules es un flag de Chromium que permite overridear la resolución DNS para dominios específicos. Le decimos: "cuando veas *.safetypanel.com.ar, resolvelo a esta IP en vez de consultar DNS". KAMAL_HOST es una variable de entorno que Kamal setea automáticamente dentro del container con la IP del host donde está corriendo. Así Chromium puede llegar al propio servidor. --ignore-certificate-errors es necesario porque el origin cert de Cloudflare está firmado por una CA de Cloudflare que Chromium no reconoce como trusted.

Todo esto se descubrió antes de tocar producción.

La VM local se pagó sola en una tarde.

Primer incendio:

El workflow de deploy necesitaba contents: read (#30)

Cuando seteás permissions explícitamente en GitHub Actions, todos los defaults se revocan.

En GitHub Actions, si no ponés el key "permissions" en el workflow YAML, todas las acciones tienen permisos de lectura por default. Pero en cuanto ponés "permissions:" para setear algo explícitamente (por ejemplo, packages: write para pushear a GHCR), TODOS los demás permisos pasan a "none". Hay que agregar explícitamente cada permiso que necesitás, incluyendo "contents: read" que normalmente viene gratis.
Feb 25

Go-Live + Primeros incendios

La app no manda emails

Como parte del QA después del deploy: entrar a la consola, disparar un mail via ActionMailer.

Timeout. Net::OpenTimeout

Testeamos puertos desde el servidor: 


ssh deploy@<vps-ip> \
  "timeout 5 bash -c '</dev/tcp/email-smtp.us-east-2.amazonaws.com/587' \
  && echo OK || echo BLOCKED"
# => BLOCKED

Puerto 587 bloqueado. Puerto 465 bloqueado. Todos los puertos SMTP bloqueados.

El comando bash '</dev/tcp/host/port' es un truco de bash para testear si un puerto TCP está accesible. No necesita instalar telnet ni netcat. Si la conexión se establece, el comando sale con exit 0 (OK). Si no, timeout lo mata después de 5 segundos (BLOCKED). DigitalOcean bloquea los puertos SMTP (25, 465, 587) por default en todos los droplets para prevenir spam. No lo desbloquean ni pidiendo — es política de la plataforma.

Contactamos al soporte de DigitalOcean:

"We are restricting SMTP traffic across the DigitalOcean platform..."
DigitalOcean Support

Pivot: reemplazamos SMTP con AWS SES API via aws-sdk-sesv2.

Deploy. Test. Funciona.

AWS SES (Simple Email Service) se puede usar de dos formas: via SMTP o via API HTTP. Como SMTP estaba bloqueado, usamos la API directamente con la gem aws-sdk-sesv2. Ya teníamos una cuenta de AWS para S3 (Active Storage), así que fue cuestión de agregar la gem, configurar el delivery method en production.rb y usar las mismas credenciales IAM.

Los jobs no reintentaban

Habiendo trabajado siempre con Sidekiq, estaba acostumbrado a que los reintentos los manejara el adapter.

Cuando migramos a Solid Queue, perdimos el esquema de reintentos.

Hubo que configurarlo explícitamente en ApplicationJob. 


class ApplicationJob < ActiveJob::Base
  retry_on StandardError, wait: :polynomially_longer, attempts: 5
end
Sidekiq tiene su propio mecanismo de retries que funciona automáticamente: por default reintenta 25 veces con backoff exponencial. Esto funciona a nivel del adapter, no de ActiveJob. Cuando pasás a Solid Queue (o cualquier otro backend que no sea Sidekiq), perdés ese comportamiento. ActiveJob tiene su propio sistema de retries con retry_on, pero hay que configurarlo explícitamente. :polynomially_longer usa la fórmula (attempt^4) + 2 segundos — es decir, 3s, 18s, 83s, 258s para los primeros 4 reintentos.

Día del deploy — pg_restore fallando, la DB managed solo accesible desde la VPC
El restore de la base de datos falló múltiples veces: 1) db:drop falló por "protected environment", 2) DISABLE_DATABASE_ENVIRONMENT_CHECK tuvo error de sintaxis, 3) pg_restore hizo timeout porque la DB managed de DigitalOcean solo es accesible desde droplets en la misma VPC, no desde la máquina local. La solución fue hacer SCP del dump al servidor y correr pg_restore desde un container Docker en el droplet, conectándose a la DB via la red interna de la VPC.
